CYBERSIREN — SYSTEM ARCHITECTURE SPECIFICATION

Document: ARCH-SPEC-v2.2 Date: 2026-03-15 Status: Implementation Baseline — Approved Architecture with Tracked Deltas Classification: Internal / Graduation Project
⚑ Spine v0 implementation deltas — This spec is the production target. The first delivered milestone (Infrastructure Spine v0, 2026-05-02) ships a local-development implementation that diverges in a few well-bounded places. Where it does, you'll see a v0 delta tag inline. Every delta is a local-dev choice that preserves the production wire contract; nothing in §1–§15 has been re-architected.
Highlights: local broker is Redpanda (Kafka-API drop-in for "Apache Kafka" in §10), retention is enforced at topic-create time per §3, all stub services emit random scores, and DB access is healthcheck-only.

1. Complete Pipeline — Step-by-Step Data Flow

Canonical Identifier Strategy:
email_id (UUIDv7) = logical identifier. Assigned by SVC-01, carried in all Kafka messages and Redis keys, used as partition key.
internal_id (BIGSERIAL) = physical DB primary key. Auto-generated on INSERT. Part of composite PK (internal_id, fetched_at) required by partitioning.
fetched_at = partition key for the emails table. Part of composite PK. Not part of logical identity.
message_id (RFC 5322) = deduplication key scoped per org (org_id, message_id). Not globally unique.
Mapping: emails.scored message includes both email_id and (internal_id, fetched_at) so SVC-08 can write to DB without an extra lookup.
Data Role Taxonomy: PostgreSQL = source of record (authoritative state). Redis = ephemeral coordination/cache (lossy, rebuildable from DB). Kafka v0 delta = transport/replay (bounded retention 24h–7d). Locally provided by Redpanda (Kafka API-compatible); see §16. Materialized Views = read optimization (never the source of truth). Object Storage (S3/MinIO) = binary retention (attachment files, 90-day TTL).

Each numbered step shows: the service performing work, the Kafka topic it produces to (if any), and the database operations (if any). Every field in every message is listed.

1
EMAIL INGESTION SERVICE (Go) — SVC-01
Receives raw email from one of 5 adapters (see §2.1). Normalizes to internal ParsedEmail envelope. Assigns email_id (UUIDv7), org_id (from API key auth), fetched_at (now). Performs idempotent dedup: checks (org_id, message_id) in Redis set (TTL 7d) → if present, discard. On cache miss, falls back to DB unique constraint uq_emails_org_message_id.
→ emails.raw
Key: email_id · 6 partitions · 48h retention
R Redis: dedup:{org_id}:{message_id}
No PostgreSQL writes at this stage.
Message schema — emails.raw:
{ email_id, org_id, fetched_at, raw_rfc822: <full email bytes base64>, source_adapter: "gmail"|"outlook"|"imap"|"api"|"custom", api_key_id }
2
EMAIL PARSER SERVICE (Go) — SVC-02
Consumes emails.raw. Parses MIME structure. Produces 5 output topics (fan-out). Extracts: all URLs from HTML (href/src/action/meta-refresh) + plain text (regex). Hashes each attachment (SHA256 + MD5 + SHA1), computes Shannon entropy, detects actual file type via magic bytes. Extracts all header authentication results. Strips HTML → clean plain text for NLP. Builds Analysis Plan listing which downstream scores to expect based on email contents. Writes initial email record + junction tables to PostgreSQL.
→ 5 topics:
analysis.urls · analysis.headers · analysis.attachments · analysis.text · analysis.plans
W emails (INSERT: 26 metadata cols from 001+002)
U enriched_threats (UPSERT bare: url, domain, tld — for email_urls FK)
W email_urls (INSERT: email_id, threat_id→enriched_threats.id, visible_text)
U attachment_library (UPSERT by sha256: sha256, md5, sha1, size_bytes, entropy)
W email_attachments (INSERT: email_id, attachment_id, filename, content_type)
W email_recipients (INSERT: email_id, org_id, address, recipient_type)

{ email_id, org_id, sender_email, sender_domain, sender_name, reply_to_email, return_path, originating_ip, x_originating_ip, auth_spf: "pass"|"fail"|"softfail"|"none", auth_dkim: "pass"|"fail"|"none", auth_dmarc: "pass"|"fail"|"none", auth_arc, mailer_agent, in_reply_to, references_list: string[], sent_timestamp: int64, content_charset, precedence, list_id, hop_count: int, received_chain: [{from, by, timestamp}], vendor_security_tags: json, headers_json: json }

{ email_id, org_id, expected_scores: ["url","header","attachment","nlp"] (subset based on email contents), url_count: int, attachment_count: int, has_body: bool, created_at: timestamp }
3a
URL ANALYSIS SERVICE (Go + Python subprocess) — SVC-03
Consumes analysis.urls. For each URL in the message:
(i) Redis cache check: key url_score:{sha256(url)} TTL 6h → if hit, reuse score, skip remaining steps.
(ii) TI feed match: lookup URL domain against ti_indicators table (indexed on indicator_value, indicator_type). If exact match with risk_score ≥ 80 → emit score 95, ti_matched=true, skip ML. Also checks enriched_threats to see if this URL was observed and enriched in a prior email.
(iii) Enrichment (concurrency limit: 5 goroutines): WHOIS (registrar, creation_date, expiry_date, name_servers), SSL cert (issuer, subject, valid_from, valid_to, self-signed check), DNS (A/MX/NS/CNAME), HTTP probe (status_code, redirect_count, final_url, response_time_ms), IP geo (country, asn, asn_name, isp, city).
(iv) ML inference: Enriched features → XGBoost model via Python subprocess (JSON stdin/stdout). Timeout: 5s. On timeout: score=50.
Per-email URL score = max(individual URL scores). Each URL gets its own entry in enrichment tables.
→ scores.url
Key: email_id · 6 partitions · 24h retention
R ti_indicators (TI feed match)
R enriched_threats (prior-email URL lookup)
R Redis: url_score:{hash}
W enriched_threats (new URLs)
W email_url_ti_matches (feed match audit trail)
W enrichment_results (raw responses)
W enrichment_jobs (job tracking)
W Redis: url_score:{hash}
Message schema — scores.url:
{ email_id, org_id, component: "url", score: int 0-100, url_count: int, ti_blocked_count: int, ml_scored_count: int, cache_hit_count: int, riskiest_url: string, url_details: [{ url, domain, ti_matched: bool, ti_source: string|null, domain_age_days: int|null, has_ssl: bool, ssl_issuer: string|null, redirect_count: int, ml_score: int|null, is_shortened: bool, enrichment_available: bool }], processing_time_ms: int }
3b
HEADER ANALYSIS SERVICE (Go) — SVC-04
Consumes analysis.headers. Produces header_risk_score (0–100) from three analysis dimensions:
(i) Authentication verification: SPF, DKIM, DMARC, ARC result evaluation. From/Reply-To/Return-Path mismatch detection.
(ii) Sender reputation: Domain age analysis (via WHOIS), TI matches on sender domain and originating IP, typosquatting detection (Damerau-Levenshtein distance to known brands), free-provider contextual signals.
(iii) Structural anomaly detection: HTML-only messages, hidden text elements, embedded forms, encoding anomalies, excessive received hops, timestamp drift.
Note: Signal categories are architecturally defined. Specific scoring weights and thresholds are preliminary heuristics subject to empirical calibration.
All signals implemented as configurable rules in the rules table. Each signal that fires creates a rule_hits row for auditability.
→ scores.header
Key: email_id · 6 partitions · 24h retention
R rules (active header rules)
R ti_indicators (domain/IP rep. via Redis cache)
W rule_hits (one per fired rule)
Message schema — scores.header:
{ email_id, org_id, component: "header", score: int 0-100, auth_sub_score: int, reputation_sub_score: int, structural_sub_score: int, fired_rules: [{ rule_id, rule_name, rule_version, score_impact: int, match_detail: json }], signals: { spf_result, dkim_result, dmarc_result, from_reply_to_match: bool, from_return_path_match: bool, domain_age_days: int|null, typosquat_target: string|null, typosquat_distance: int|null, is_free_provider: bool, hop_count: int, time_drift_hours: float }, processing_time_ms: int }
3c
ATTACHMENT ANALYSIS SERVICE (Go) — SVC-05
Consumes analysis.attachments. For each attachment:
(i) Hash lookup: attachment_library.sha256 → if is_malicious=true, score=90.
(ii) External TI check (if VirusTotal API key configured): query hash → cache in enrichment_results provider="virustotal". TTL: 24h.
(iii) Heuristics: entropy >7.5=+20. Extension vs MIME mismatch (e.g. .pdf with application/x-msdownload)=+30. Dangerous extensions (.exe,.scr,.bat,.ps1,.vbs,.js,.wsf,.hta,.cmd)=+25. Macro-enabled Office (.docm,.xlsm,.pptm)=+20. Double extension (.pdf.exe)=+35.
Per-email attachment score = max(individual attachment scores).
→ scores.attachment
Key: email_id · 3 partitions · 24h retention
R attachment_library (hash lookup)
R/W enrichment_results (VT cache)
W attachment_library (update is_malicious, risk_score, threat_tags)
Message schema — scores.attachment:
{ email_id, org_id, component: "attachment", score: int 0-100, attachment_count: int, malicious_count: int, attachment_details: [{ sha256, filename, content_type, size_bytes, entropy: float, is_malicious: bool, ti_source: string|null, extension_mismatch: bool, has_macros: bool, is_dangerous_extension: bool, individual_score: int }], processing_time_ms: int }
3d
NLP ANALYSIS SERVICE (Python FastAPI + ONNX Runtime) — SVC-06
Consumes analysis.text. Produces content_risk_score (0–100) and structured facet outputs from 4 analysis dimensions:
(i) Urgency detection: Identifies false-urgency language patterns. Output: urgency_score 0.0–1.0.
(ii) Intent classification: Classifies email goal. Multi-class: credential_harvesting, malware_delivery, bec, scam, legitimate. Output: intent_label + confidence.
(iii) Brand impersonation: Detects brand impersonation through NER + sender identity analysis. Output: impersonation_score 0.0–1.0.
(iv) Deception language: Identifies manipulative linguistic patterns. Output: deception_score 0.0–1.0.
Note: Analysis facets are architecturally defined. Model architectures (candidate: DistilBERT, TinyBERT), training methodology, and score composition method are under active research.
Individual facet scores are always preserved alongside any composite score for transparency.
Timeout: 10s. On timeout: score=50. Stateless — no DB access.
→ scores.nlp
Key: email_id · 6 partitions · 24h retention
No database access.
Stateless inference service.
Message schema — scores.nlp:
{ email_id, org_id, component: "nlp", score: int 0-100, facets: { urgency_score: float, intent_label: "credential_harvesting"|"malware_delivery"|"bec"|"scam"|"legitimate", intent_confidence: float, impersonation_score: float, impersonated_brand: string|null, deception_score: float }, model_versions: { urgency: string, intent: string, impersonation: string, deception: string }, processing_time_ms: int }
4
SCORE AGGREGATOR SERVICE (Go + Redis state) — SVC-07
Consumes 5 topics: analysis.plans + scores.url + scores.header + scores.attachment + scores.nlp.
Maintains per-email state in Redis: Key: aggregator:{email_id} — TTL 120s.
Design: Stateless workers with shared Redis state. Events for the same email may be processed by different instances. Redis is the single source of truth for aggregation state, not instance-local memory.
Function: Gather/synchronization barrier only — does NOT perform scoring. Collects all component scores and packages them into a single payload for the Decision Engine.
Completion check: After each score received, compare received keys to expected list. All present → package and publish to emails.scored. Not all → wait.
Timeout: 30s → emit partial result with partial_analysis: true and list of missing components.
→ emails.scored
Key: email_id · 6 partitions · 48h retention
R/W Redis only: aggregator:{email_id}
No PostgreSQL access.
Message schema — emails.scored:
{ email_id, org_id, internal_id: bigint, fetched_at: timestamp, url_score: int|null, header_score: int, attachment_score: int|null, nlp_score: int|null, partial_analysis: bool, missing_components: string[], component_details: { url: <full scores.url message>, header: <full scores.header message>, attachment: <full scores.attachment message>|null, nlp: <full scores.nlp message>|null }, aggregation_latency_ms: int, timeout_triggered: bool }
5
DECISION ENGINE / VERDICT SERVICE (Go) — SVC-08
Consumes emails.scored. Terminal processing block — responsible for final scoring, verdict, campaign management, and persistence:
(i) Final score computation: Receives individual component scores (url, header, attachment, nlp) from Aggregator. Combines into a single risk_score (0–100). Combination method is under research — must be explainable; individual component scores always preserved.
(ii) Rule engine: Load active rules from rules table (60s cache). Each firing rule adjusts score by score_impact (-100 to +100).
(iii) Verdict: Map final score to label. Preliminary thresholds (subject to calibration): 0–25 → benign | 26–50 → suspicious | 51–75 → phishing | 76–100 → phishing (high) or malware. Confidence computed separately from risk score — reflects label certainty, not severity.
(iv) Campaign management: fingerprint = SHA256(sender_domain|url_domain|subject_template|intent). UPSERT into campaigns. SimHash near-duplicate detection (per-org scoped via Redis).
(v) Campaign-informed scoring: Campaign history (avg risk_score, email count, analyst verdicts) serves as an additional signal for future emails matching the same campaign. Specific mechanism under research.
(vi) All DB writes within a single PostgreSQL transaction.
→ emails.verdict
Key: email_id · 6 partitions · 7d retention
R rules (003+008) · R campaigns (001+002) · R verdicts (campaign history) · R Redis: simhash:{org_id}:*
W emails UPDATE: risk_score, header_risk_score, content_risk_score, url_risk_score, attachment_risk_score, campaign_id, analysis_metadata
W verdicts INSERT: entity_type, entity_id, label, confidence, source, model_version
W rule_hits: rule_id, entity_id, score_impact, match_detail
U campaigns: fingerprint, org_id, threat_type, target_brand, risk_score
W Redis: simhash:{org_id}:{campaign_id}
Message schema — emails.verdict:
{ email_id, org_id, verdict_label: "benign"|"suspicious"|"phishing"|"malware"|"spam"|"unknown", confidence: float 0-1, risk_score: int 0-100, header_risk_score: int, content_risk_score: int, url_risk_score: int, attachment_risk_score: int, campaign_id: int|null, campaign_fingerprint: string|null, is_new_campaign: bool, fired_rules: [{ rule_id, rule_name, score_impact }], verdict_source: "model"|"rule", model_version: string, partial_analysis: bool, processing_time_total_ms: int }

8. Campaign Detection Algorithm

8.1 Campaign Fingerprint Computation (Exact Match)

INPUT: scored email with verdict

  sender_domain  = lower(email.sender_domain)                           // e.g. "paypa1.com"
  url_domain     = lower(domain_of(email.riskiest_url))                 // e.g. "secure-login.paypa1.com"
                   OR "" if no URLs
  subject_tpl    = template(email.subject)                              // replace names/nums with placeholders
                   e.g. "Your {BRAND} account #{NUM} needs verification"
  intent         = nlp_facets.intent_label                              // e.g. "credential_harvesting"

  fingerprint    = SHA256( sender_domain + "|" + url_domain + "|" + SHA256(subject_tpl) + "|" + intent )

DB OPERATION:
  INSERT INTO campaigns (fingerprint, org_id, threat_type, target_brand, first_seen, last_seen, risk_score)
  VALUES ($fingerprint, $org_id, $threat_type, $target_brand, NOW(), NOW(), $risk_score)
  ON CONFLICT (fingerprint) DO UPDATE SET
      last_seen = NOW(),
      risk_score = (campaigns.risk_score + $risk_score) / 2;   -- rolling average

  UPDATE emails SET campaign_id = <returned campaign_id> WHERE internal_id = $email_id AND fetched_at = $fetched_at;

8.2 SimHash Near-Duplicate Detection (Fuzzy Match)

INPUT: email plain_text body

  simhash = compute_simhash_64bit(tokenize(plain_text))       // 64-bit SimHash
  
  FOR each campaign_id in Redis keys matching simhash:{org_id}:*:
      stored_hash = GET simhash:{org_id}:{campaign_id}
      distance = hamming_distance(simhash, stored_hash)
      IF distance <= 3:                                        // threshold: 3 bits out of 64
          → associate email with this campaign (even if fingerprint differs)
          → BREAK (take first match; campaigns ordered by last_seen DESC)
  
  IF no match found AND email.risk_score >= 50:
      → new campaign was already created by fingerprint logic above
      → SET simhash:{org_id}:{campaign_id} = simhash (TTL 30d)

Redis 7+

  • 8 key patterns (see §5 for full registry)
  • Persistence: AOF with 1s fsync
  • Max memory: 2 GB (LRU eviction on TI cache keys)

  • Recommended: 2 vCPU, 4 GB RAM

Design note — email_urls.threat_id FK dependency:
The email_urls table (001) has a threat_id BIGINT NOT NULL REFERENCES enriched_threats(id). This means the Email Parser (SVC-02) must UPSERT a bare row into enriched_threats (with just url + domain) before it can INSERT into email_urls, to obtain the threat_id FK. The URL Analysis Service (SVC-03) later UPDATEs that same enriched_threats row with full enrichment data (WHOIS, SSL, geo, etc.). This two-phase write pattern is documented in the column-level flow below.

Design note — email_url_ti_matches junction table (026):
When SVC-03 matches a URL against ti_indicators (feed data), it records the match in email_url_ti_matches linking email_urls.id → ti_indicators.id. This separates the feed-match audit trail from the enrichment data in enriched_threats. Feed indicators (ti_indicators) are cheap, normalized, and used for matching only. Email-observed threats (enriched_threats) carry expensive enrichment data (WHOIS/SSL/geo/ASN).

16. Changelog — Implementation Deltas

This section records, in reverse-chronological order, every place where a delivered milestone diverged from the baseline spec above. Each entry names the spec section it amends, the change, and the rationale. The baseline text in §1–§15 is intentionally left unchanged so the production target remains visible alongside the local-dev reality.

Infrastructure Spine v0

spine-v02026-05-02 — First end-to-end backbone: a fake email flows emails.raw → emails.verdict through 10 stub services with traces, metrics, and logs.
Δ Affects Baseline (spec) v0 implementation Rationale
D1 §10 Apache Kafka Apache Kafka, 3 brokers, RF 3, ISR min 2, ZK or KRaft. Single-node Redpanda (KRaft, dev-container) on the same Kafka API; RF 1. Local dev. Redpanda is wire-compatible with Apache Kafka, so the Go client (github.com/twmb/franz-go), topic registry (§3), partition strategy, consumer groups (§3 right column), and trace-header propagation (§12) are unchanged. Single binary boots in ~3 s vs. ~25–40 s for ZK + Kafka, idles at ~200 MB instead of ~1.6 GB, and removes the ZK container entirely. Production deployment can swap the broker image back to Apache Kafka without touching application code.
D2 §3 Kafka Topic Registry Per-topic retention "documented" but not enforced at create time. Retention applied at topic create via rpk topic create --topic-config retention.ms=…. emails.raw=48 h, analysis.* & scores.*=24 h, emails.scored=48 h, emails.verdict=7 d. cleanup.policy=delete, compression.type=zstd. Topic registry numbers are now load-bearing rather than informational. zstd halves on-disk footprint with negligible CPU cost on the dev box.
D3 §10 Exactly-once Exactly-once semantics: idempotent producers + transactional consumers. Idempotent producer enabled (kgo.AllISRAcks(), kgo.RequiredAcks); transactional consumer deferred. Manual commit on handler success. Spine v0's smoke test does not exercise multi-record transactions, so the simpler at-least-once + dedup approach is sufficient for the milestone. Transactional consumer will be added when the first real (non-stub) writer service is implemented.
D4 §1, §3 Wire format Spec lists fields per topic but doesn't pin a serialisation. JSON with Go structs in shared/contracts/kafka/; envelope carries email_id, org_id, timestamp, trace_id, schema_version. JSON is debuggable from the CLI (rpk topic consume --format '%v') and matches the field-by-field shape of the spec. schema_version on the envelope leaves room to introduce a Schema Registry later without breaking existing consumers.
D5 §4 Per-service stubbing 11 services with full business logic. 8 services run as stubs: parser fans out a copy of the meta envelope; analysers (svc-03 url-pipeline, svc-04, svc-05, svc-06 nlp-pipeline) emit score = rand(0,100); aggregator gathers in Valkey hash aggregator:{email_id}; decision averages and threshold-maps; svc-09/10 are sinks. Existing real svc-03 /scan, svc-06 /predict, and svc-11 TI-sync demos are untouched. v0's goal is to prove the backbone, not the analysers. Random scores still exercise the threshold mapping in §1 step 5 (verdict can land in any of benign / suspicious / phishing / malware). Pipeline-side binaries are new (cmd/url-pipeline, cmd/nlp-pipeline) so the existing HTTP demos under cmd/url-analysis and cmd/nlp remain runnable in parallel via make demo.
D6 §13 Database access patterns Every service reads/writes its assigned tables (see §13). Healthcheck-only. Each stub does pool.Ping() at boot and never INSERT/UPDATE/SELECT. internal_id and fetched_at on emails.scored / emails.verdict are placeholders ("fake-internal-{email_id}", time.Now()). Aligning logical email_id to physical (internal_id, fetched_at) requires SVC-01 to write the emails row, which is out of scope for v0. The placeholder values flow through the contract types unmodified, so when SVC-01 starts writing for real the downstream services need no changes.
D7 §5 Redis aggregator key aggregator:{email_id} hash, TTL 120 s; emit on completeness or 30 s timeout. Hash + TTL implemented. Timeout-driven emit is deferred; v0 emits only on "all expected scores arrived". The smoke test always produces every expected score (4 random analysers always emit), so the timeout path is never reached in v0. Adding it now would be untested code; defer to the milestone that produces real intermittent score topics.
D8 §12 Observability OpenTelemetry, Kafka header trace propagation (traceparent), email_id on every span, Prometheus per service. Implemented as specified, via the kotel plugin for franz-go. Three new pipeline-wide counters live in shared/kafka/metrics.go: cybersiren_kafka_messages_consumed_total, …produced_total, cybersiren_pipeline_processing_seconds. Verified by Jaeger showing one trace ID with spans across all 10 services. This is the spine v0 acceptance evidence — see demo-infra.md.
D9 §2 Service skeletons (implicit) — every service has its own bootstrap. New shared package shared/svckit/ wraps config + logger + tracing + metrics/healthz + Postgres ping + Valkey + Kafka producer/consumer + graceful shutdown. Each stub main.go is ~30 lines. Reduces duplicated bootstrap code and ensures every service initialises observability the same way. Real services can opt out of any field (e.g. svc-09 has no producer).
D10 §14 Deployment (production-style) Compose profiles for kafka, postgres, valkey, observability. The kafka compose profile name is preserved; behind it: redpanda + redpanda-console + kafka-init (renamed mentally to "topic-init" but the file is still kafka-init.sh). Existing make targets (make up, make down, make kafka-topics) work unchanged. Keeps the developer's mental model and muscle memory intact while swapping the implementation underneath.

Validation evidence: see demo-infra.md at the repo root for the operator's view (smoke run, trace verification, Prometheus counters showing 1:1 conservation across the DAG).
