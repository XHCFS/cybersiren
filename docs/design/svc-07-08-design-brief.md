## 1. Full Pipeline Context

### 1.1 Architecture Overview

CyberSiren is a Kafka-based phishing detection pipeline. Emails enter at SVC-01 and flow through 10 stages. SVC-07 and SVC-08 are the final two processing stages before notification.

```
SVC-01 (ingestion)
  → emails.raw
    SVC-02 (parser)    ← fan-out to 5 topics
      → analysis.urls       → SVC-03 (URL classifier + TI blocklist) → scores.url
      → analysis.headers    → SVC-04 (header/SPF/DKIM/DMARC rules)  → scores.header
      → analysis.attachments→ SVC-05 (attachment heuristics)        → scores.attachment
      → analysis.text       → SVC-06 (DistilBERT NLP)               → scores.nlp
      → analysis.plans      ──────────────────────────────────────────────────────┐
                                                                                   ▼
                                                              SVC-07 (aggregator) ← all 5 above
                                                                → emails.scored
                                                                  SVC-08 (decision engine)
                                                                    → emails.verdict
                                                                      SVC-09 (notification)
                                                                      SVC-10 (API/dashboard)
```

### 1.2 Technology Stack

- **Language:** Go 1.25
- **Message broker:** Redpanda (Kafka-API compatible, local dev; Apache Kafka in prod)
- **Kafka client library:** `github.com/twmb/franz-go` (NOT kafka-go, NOT sarama)
- **Database:** PostgreSQL 15 (accessed via `github.com/jackc/pgx/v5`)
- **In-memory store:** Valkey (Redis-compatible; client: `github.com/valkey-io/valkey-go`)
- **Metrics:** Prometheus (`github.com/prometheus/client_golang`)
- **Tracing:** OpenTelemetry → Jaeger (`go.opentelemetry.io/otel`)
- **Structured logging:** zerolog (`github.com/rs/zerolog`)
- **Service harness:** `shared/svckit` — handles config, logger, Kafka consumers/producers, Postgres pool, Valkey client, Prometheus server, graceful shutdown. **Both services use this.**

### 1.3 Module Path

All imports use `github.com/saif/cybersiren/...`.

### 1.4 Key Identifiers

The pipeline uses **three** identifier concepts — these are critical for SVC-08's DB writes:

| Name | Type | Meaning | Source |
|------|------|---------|--------|
| `email_id` | `int64` | Logical identifier, carries on all Kafka messages and Redis keys | Generated by SVC-01 |
| `internal_id` | `int64 BIGSERIAL` | Physical DB primary key on `emails` table | Auto-generated by Postgres on `INSERT` |
| `fetched_at` | `time.Time` | Partition key on `emails` (monthly range partitions) | Set at ingest time |

`emails.scored` carries **both** `email_id` (logical) and `(internal_id, fetched_at)` (physical DB key) so SVC-08 can write to Postgres without doing an extra lookup. This is a deliberate architectural decision in ARCH-SPEC §1.

## 2. SVC-07 — Score Aggregator

### 2.1 Role

SVC-07 is a **pure synchronization barrier**. It does **no scoring**. Its only job is to wait until all expected component scores for one email have arrived, then package them into a single `emails.scored` message and emit it to SVC-08.

It is **stateless** across instances: all per-email state lives in Valkey. Multiple SVC-07 instances can run in parallel; messages for the same email may arrive at different instances.

### 2.2 Inputs

Consumes **5 Kafka topics** in consumer group `cg-aggregator`:

| Topic | Producer | Shape |
|-------|----------|-------|
| `analysis.plans` | SVC-02 | `AnalysisPlan` — declares which score topics to expect |
| `scores.url` | SVC-03 | `ScoreEnvelope` (generic) |
| `scores.header` | SVC-04 | `ScoresHeaderMessage` (flat, **different shape from others**) |
| `scores.attachment` | SVC-05 | `ScoreEnvelope` (generic) |
| `scores.nlp` | SVC-06 | `ScoreEnvelope` (generic) |

**The plan message is critical.** SVC-02 publishes to `analysis.plans` simultaneously with the other topics. It contains `expected_scores: []string` — the list of score-topic names the aggregator should wait for (e.g., `["scores.url", "scores.header", "scores.nlp"]` — attachment may be absent for emails with no attachments). The aggregator must wait only for topics in this list.

### 2.3 Output

Produces to `emails.scored` (consumer group for downstream: `cg-decision-engine`).

**Full required emails.scored wire schema** (this is what SVC-08 reads):
```json
{
  "email_id": 1234567890,
  "org_id": 1,
  "internal_id": 42,
  "fetched_at": "2026-05-03T10:00:00Z",
  "url_score": 72,
  "header_score": 85,
  "attachment_score": null,
  "nlp_score": 60,
  "partial_analysis": false,
  "missing_components": [],
  "component_details": {
    "url": { /* full scores.url message */ },
    "header": { /* full scores.header message (ScoresHeaderMessage) */ },
    "attachment": null,
    "nlp": { /* full scores.nlp message */ }
  },
  "aggregation_latency_ms": 312,
  "timeout_triggered": false
}
```

**Important:** `component_details` carries the **complete score messages**, not just the score number. SVC-08 needs `component_details.header` to extract `sender_domain` for campaign fingerprinting. This is why the full message must be forwarded, not just the score value.

**Note on the current contract:** `shared/contracts/kafka/emails.go` defines `EmailsScored` with `ComponentScores map[string]float64`. This is the v0 stub shape. The real implementation needs this contract extended to carry the full schema above. **The contract change must be backward-compatible or versioned** — discuss the approach in your design.

### 2.4 Valkey State

- **Key:** `aggregator:{email_id}` (e.g., `aggregator:1234567890`)
- **Type:** Redis Hash
- **TTL:** 120 seconds from most recent write (refreshed on each `HSET`)
- **Fields:**
  - `__plan` → raw JSON of `AnalysisPlan` (set when `analysis.plans` message arrives)
  - `scores.url` → raw JSON of scores.url message
  - `scores.header` → raw JSON of scores.header message
  - `scores.attachment` → raw JSON (set only if attachment scores arrive)
  - `scores.nlp` → raw JSON
- **Completion check:** `HGETALL`, parse `__plan`, check all `expected_scores` keys are present

### 2.5 The 30-Second Timeout (MISSING FROM STUB — MUST IMPLEMENT)

The stub comment says explicitly: `// NO 30-second timeout in v0`. This **must** be implemented.

**Mechanism:** After a score arrives, check completion. If not complete, also check age:
- Compute `first_message_time` — store the `time.Now()` when the **first** message for this `email_id` arrives (use a dedicated hash field, e.g., `__started_at`).
- If elapsed since `__started_at` ≥ 30 seconds AND not all scores present → emit partial result with `partial_analysis: true`, list of `missing_components`, `timeout_triggered: true`.
- Delete the Valkey key after emitting.

**Design question:** How to trigger the timeout check. Two options:
1. **Piggyback:** On each new score arrival, check both completion AND age. If timed-out, emit partial. Simple but only fires when a message arrives (an email with only 1 score arriving and then nothing will never emit).
2. **Background sweeper:** A separate goroutine polls Redis for stale keys and emits partial results. More correct but more complex.

The spec implies option 2 is needed for correctness. Design and propose the right approach.

### 2.6 At-Least-Once Safety

The Kafka consumer is at-least-once. The current stub deletes the Valkey key **after** publishing. If the delete fails, the email will be aggregated again on the next score arrival. This is acceptable (idempotent UPSERT at SVC-08). But if the **publish succeeds** and the **delete fails**, SVC-08 will receive duplicates. Design must address this.

### 2.7 Message Order Considerations

`analysis.plans` may arrive **after** some score messages (Kafka does not guarantee cross-topic ordering). The aggregator must handle this: store whatever arrives first, check completion only once the plan is present, and re-check completion for all previously stored scores when the plan arrives.

The current stub already handles this correctly (check `all[planField]` before checking completion).

### 2.8 What the Stub Already Has (Keep These)

The existing `services/svc-07-aggregator/cmd/aggregator/main.go` (180 lines) already correctly implements:
- `flexShape` decoder that handles both `ScoresHeaderMessage` (flat int64 email_id) and `ScoreEnvelope` (nested MessageMeta)
- `HSET` + `EXPIRE` + `HGETALL` pattern
- Plan-based completion check
- Publish and DEL on completion
- `decodeScore()` that normalises scores from both shapes

**Do not throw these away.** The real implementation should promote this logic into `internal/aggregator/` and add the missing pieces on top.

### 2.9 Required Prometheus Metrics

Mirror the SVC-04 pattern (`internal/processor/metrics.go`):
- `aggregator_messages_total{topic, status}` — per-topic message counts
- `aggregator_completion_latency_ms` — histogram of time from first message to completion
- `aggregator_partial_completions_total` — counter when timeout triggers partial emit
- `aggregator_active_buckets` — gauge of how many email_ids are currently in-flight (Valkey keys)

## 3. SVC-08 — Decision Engine

### 3.1 Role

SVC-08 is the **terminal processing block**. It receives the aggregated score package from SVC-07 and:
1. Computes the final `risk_score` (0–100) by blending component scores
2. Applies the rule engine to adjust the score
3. Maps the score to a verdict label
4. Computes a confidence value
5. Manages campaign fingerprinting and deduplication
6. Writes everything to Postgres in a **single transaction**
7. Emits `emails.verdict` to Kafka

### 3.2 Input

Consumes `emails.scored` in consumer group `cg-decision-engine`.

### 3.3 Output

Produces to `emails.verdict` (7-day retention).

**Full required emails.verdict wire schema:**
```json
{
  "email_id": 1234567890,
  "org_id": 1,
  "verdict_label": "phishing",
  "confidence": 0.82,
  "risk_score": 78,
  "header_risk_score": 85,
  "content_risk_score": 60,
  "url_risk_score": 72,
  "attachment_risk_score": null,
  "campaign_id": 17,
  "campaign_fingerprint": "a3f8...",
  "is_new_campaign": false,
  "fired_rules": [
    {"rule_id": 42, "rule_name": "spf-fail-high-entropy-subject", "score_impact": 15}
  ],
  "verdict_source": "model",
  "model_version": "xgb-v1.2+distilbert-v1.0",
  "partial_analysis": false,
  "processing_time_total_ms": 45
}
```

### 3.4 Score Blending (Step i)

Receives `url_score`, `header_score`, `attachment_score` (nullable), `nlp_score` (nullable) from SVC-07.

**Current stub:** simple average — `sum / len(component_scores)`.

**Required:** Configurable weighted blend. Starting weights (equal by default, env-var overrides):
- `url_weight` = 0.35 (URL is usually the strongest signal)
- `header_weight` = 0.30
- `nlp_weight` = 0.25
- `attachment_weight` = 0.10

**Null components:** If `partial_analysis: true` in emails.scored, some scores may be missing. The blender must handle null scores gracefully — divide by the sum of weights of **present** components only, not total. Flag this condition in the output (`partial_analysis: true` passes through).

**Explainability invariant:** Individual component scores must be preserved in the `emails.verdict` message and in the `emails` DB UPDATE regardless of the blending method.

### 3.5 Rule Engine (Step ii)

The DB contains a `rules` table. Each rule has:
- `target`: the kind of entity the rule evaluates (`'email'`, `'url'`, `'header'`, `'attachment'`)
- `logic`: JSONB blob in the same JSON-DSL already implemented in `services/svc-04-header-analysis/internal/rules/` (see §4.2)
- `score_impact`: integer −100 to +100, added to the blended score
- `status`: only `'active'` rules fire

SVC-08 loads rules with `target IN ('email', 'decision')` (SVC-04 loads `target IN ('header', 'email')`). The overlap is intentional — `target='email'` rules fire in both places.

**Cache:** Identical 3-tier pattern to SVC-04: in-memory (atomic.Value per org_id) → Valkey `rules_cache:{org_id}` → Postgres. TTL: 60 seconds. **Reuse `services/svc-04-header-analysis/internal/rules/cache.go` as the template** — this is nearly copy-pasteable with a different `Targets` list. Do not rewrite from scratch.

**Rule evaluation signal snapshot for SVC-08** (passed to the DSL evaluator):
```
"score.url"          → float64 (url_score from emails.scored)
"score.header"       → float64
"score.nlp"          → float64
"score.attachment"   → float64 (or absent if null)
"score.blended"      → float64 (the pre-rule blended score)
"partial_analysis"   → bool
"verdict.label"      → string (pre-rule verdict label, before adjustments)
"campaign.is_new"    → bool
"campaign.risk_score"→ float64 (rolling average from campaigns table, 0 if new)
"campaign.email_count"→ int
```

Each fired rule adds its `score_impact` to the blended score (clamped to [0, 100]).  
Each fired rule is written to `rule_hits` with `entity_type='email'`, `entity_id=internal_id`.

**The DSL evaluator already exists** — see `services/svc-04-header-analysis/internal/rules/dsl.go` and `evaluator.go`. SVC-08 can import those directly (they operate on `SignalSnapshot = map[string]any`). The evaluator returns `EvaluationResult` with `Fired []FiredRule`.

### 3.6 Verdict Thresholds (Step iii)

After score blending + rule adjustments:

| Score range | Label |
|-------------|-------|
| 0–25 | `benign` |
| 26–50 | `suspicious` |
| 51–75 | `phishing` |
| 76–100 | `malware` |

Note: `spam` and `unknown` labels exist in the `verdict_label` enum but are not yet assigned by the automated pipeline (reserved for analyst overrides).

### 3.7 Confidence Score (Step iii, continued)

**Confidence reflects label certainty, NOT severity.** A score of 13 is just as high-confidence "benign" as a score of 98 is "malware".

**Formula from issue #96:**
```
base_confidence = distance_from_nearest_threshold / 25
```
Where `distance_from_nearest_threshold` = minimum of:
- `abs(score - lower_threshold_of_label)`
- `abs(score - upper_threshold_of_label)`

For a score of 78 (malware, range 76–100): `min(78-76, 100-78) = min(2, 22) = 2`. `base = 2/25 = 0.08`. Low confidence — barely in the malware band.

For a score of 98: `min(98-76, 100-98) = min(22, 2) = 2`. Same: `0.08`. Also low — near the edge.

For a score of 88: `min(88-76, 100-88) = min(12, 12) = 12`. `base = 12/25 = 0.48`. Moderate.

**Penalty multipliers:**
- If `partial_analysis: true`: multiply by 0.7 (missing scores reduce certainty)
- If `verdict_source == "rule"` (no ML scores, rule-only): multiply by 0.5

Clamp result to [0.0, 1.0].

**`verdict_source` logic:** If the pipeline produced full ML scores (url + header + nlp), `source = "model"`. If only rule-engine scores contributed (all ML scores were null/missing), `source = "rule"`.

### 3.8 Campaign Management (Step iv + v)

#### 3.8.1 Campaign Fingerprint

```
fingerprint = SHA256( sender_domain + "|" + url_domain + "|" + SHA256(subject_template) + "|" + intent )
```

Fields sourced from `emails.scored.component_details`:
- `sender_domain`: from `component_details.header.signals.sender_domain` OR `component_details.header` message
- `url_domain`: the **primary** URL domain from `component_details.url` (if present), else empty string
- `subject_template`: the email subject with variable content normalized (see §3.8.2)
- `intent`: from NLP output — `component_details.nlp.details["intent"]` (e.g., `"phishing"`, `"urgency"`, `""`)

**Missing fields:** If any field is unavailable (partial analysis, null component), substitute `""` (empty string). The fingerprint is still computed and stored; a partial fingerprint is better than no campaign tracking.

#### 3.8.2 Subject Template Normalization

The raw subject may contain unique tokens (order IDs, transaction IDs, dates) that would prevent matching related phishing emails. Normalize by:
1. Convert to lowercase
2. Remove leading `Re:`, `Fwd:`, `[EXTERNAL]`, and similar prefixes
3. Replace sequences of digits (≥3) with `{N}` (e.g., `ORDER-1234567` → `ORDER-{N}`)
4. Replace UUID patterns with `{UUID}`
5. Replace email addresses with `{EMAIL}`

The normalized form is **not stored** — it is used only to compute the inner `SHA256(subject_template)`, which is then concatenated into the outer fingerprint.

#### 3.8.3 UPSERT into Campaigns

```sql
INSERT INTO campaigns (fingerprint, org_id, name, threat_type, target_brand, first_seen, last_seen, risk_score, tags)
VALUES ($1, $2, $3, $4, $5, now(), now(), $6, $7)
ON CONFLICT (fingerprint) DO UPDATE SET
  last_seen  = now(),
  risk_score = ROUND((campaigns.risk_score * campaigns_email_count + EXCLUDED.risk_score) / (campaigns_email_count + 1)),
  ...
RETURNING id, (xmax = 0) AS is_new
```

The `risk_score` rolling average on campaigns is a running mean. The exact UPDATE formula needs a separate `email_count` column or the UPDATE must use a subquery — design accordingly.

Current `campaigns` table columns (from migration 001):
```
id, name, description, fingerprint, threat_type, target_brand, tags,
first_seen, last_seen, risk_score, analysis_metadata, deleted_at,
created_at, updated_at, org_id, brand_id
```
**Note:** There is no `email_count` column on campaigns. Rolling average requires knowing the current count. Use `analysis_metadata JSONB` to store `{"email_count": N}` for the running average, or propose adding the column via a new migration.

#### 3.8.4 SimHash Near-Duplicate Detection

The spec mentions `SimHash near-duplicate detection (per-org scoped via Redis)`. This is the mechanism to detect when two emails belong to the same campaign **even if their fingerprint is slightly different** (e.g., different URL domain due to typosquatting). The exact algorithm is flagged as a research question — see §6.2.

### 3.9 Campaign-Informed Scoring (Step v — research needed)

The spec says: "Campaign history (avg risk_score, email count, analyst verdicts) serves as an additional signal for future emails matching the same campaign. Specific mechanism under research."

Concretely: if `campaigns.risk_score` for the matched campaign is significantly higher than the current blended score, the campaign history should nudge the score upward. See §6.3 for the research prompt.

### 3.10 Database Writes — Single Transaction (Step vi)

All of the following happen inside **one** `pgx.TxOptions{}` transaction. Any failure rolls back everything. Kafka offset is NOT committed on failure.

**Writes (in order within the transaction):**

1. **UPDATE emails** (set score columns and campaign_id):
   ```sql
   UPDATE emails
   SET risk_score            = $3,
       header_risk_score     = $4,
       content_risk_score    = $5,
       url_risk_score        = $6,
       attachment_risk_score = $7,
       campaign_id           = $8,
       analysis_metadata     = $9
   WHERE internal_id = $1 AND fetched_at = $2
   ```
   Uses the composite PK `(internal_id, fetched_at)` from `emails.scored`. This is required for the partitioned table.

2. **INSERT INTO verdicts** (append-only, never update):
   ```sql
   INSERT INTO verdicts (entity_type, entity_id, email_fetched_at, label, confidence, source, model_version, org_id)
   VALUES ('email', $1, $2, $3, $4, $5, $6, $7)
   RETURNING id
   ```
   `entity_id = internal_id`, `email_fetched_at = fetched_at`. Both required for partitioned table joins.

3. **INSERT INTO rule_hits** (one row per fired rule):
   Reuse `db/sqlc/rules.sql.go:InsertRuleHit`. See `services/svc-04-header-analysis/internal/processor/writer.go` for the exact pattern with retry-with-backoff.

4. **UPSERT campaigns** (see §3.8.3).

5. **UPDATE emails.campaign_id** (if campaign was just created or updated, link back). This may already be covered in step 1 if campaign UPSERT RETURNING id provides the ID before the emails UPDATE.

**Transaction ordering:** Campaign UPSERT must run before emails UPDATE (need the campaign ID). Rule_hits can run after emails UPDATE.

### 3.11 Missing sqlc Queries

The following queries **do not yet exist** in `db/queries/` and need to be added:
- `UpdateEmailScores` — UPDATE emails with risk scores and campaign_id
- `InsertVerdict` — INSERT INTO verdicts RETURNING id  
- `UpsertCampaign` — INSERT ... ON CONFLICT (fingerprint) DO UPDATE ... RETURNING id, is_new

After adding `.sql` files, run `sqlc generate` to produce Go code in `db/sqlc/`.

### 3.12 Required Prometheus Metrics

- `decision_messages_total{status}` — processed/error
- `decision_risk_score` — histogram of final risk scores (buckets: 0,25,50,75,100)
- `decision_verdict_total{label}` — per-label verdict counts
- `decision_campaign_total{type}` — new/existing campaign
- `decision_db_write_duration_seconds` — histogram of transaction duration
- `decision_rules_fired_total{rule_id}` — per-rule counter
- `decision_processing_duration_seconds` — end-to-end per-message latency

## 5. Open Design Questions

These questions are flagged in the spec as "under research" or have trade-offs that require deliberate choices. Answers from research in §6 should inform the design.

### Q1 — Score Blending Method
**Status:** Spec says "combination method is under research — must be explainable."  
**Current stub:** simple average.  
**Constraint:** Individual component scores must be preserved (not lossy).  
**Options:** weighted average (simplest, explainable), log-odds blending (Bayesian, less interpretable), max-score (conservative, high recall), rank aggregation, calibrated ensemble.

## 8. Constraints and Non-Negotiables

1. **Single DB transaction** in SVC-08. No exceptions. All 4 write operations (emails UPDATE, verdicts INSERT, rule_hits INSERT, campaigns UPSERT) in one `pgx.BeginTx` / `Commit`.
2. **Kafka offset not committed** on DB write failure. Return `error` from the handler.
3. **Malformed Kafka messages** → commit offset (return nil), log error, increment error metric. Do not block the partition forever on poison pills.
4. **`verdicts` table is append-only.** Never UPDATE a row. Always INSERT.
5. **`rule_hits` table is append-only.** Same.
6. **SVC-07 has zero Postgres access.** Only Valkey.
7. **Both services must expose Prometheus metrics** and OpenTelemetry traces on the existing ports (configured by svckit).
8. **No new top-level dependencies** beyond what is already in `go.mod`. If SimHash requires a small library, add it; but do not pull in a full ML framework or ONNX runtime.
9. **The DSL evaluator from svc-04** must be reused for rule evaluation in SVC-08 — do not reimplement it.
10. **All scores passed as `int` in the verdict Kafka message**, as `float64` internally for blending (matching the existing ScoreEnvelope.Score type).
