Document: NLP-SVC-SPEC-v1.0 Date: 2026-04-21 Scope: SVC-06 NLP — Implemented Go Wrapper + Python Inference Service Classification: Internal / Graduation Project
Context: This document specifies the implemented SVC-06 NLP service. It covers the dual-service runtime architecture (Go HTTP wrapper + Python FastAPI inference engine), the complete inference pipeline from raw email text to classification verdict, the API contract, observability instrumentation, configuration, and deployment artifacts. For the ML model training methodology and research basis, see CyberSiren_NLP_Email_Classification_Model_Specification.html. This companion document describes what runs in production.
Key facts: Model = DistilBERT INT8 ONNX (66.8 MB)  |  Output classes = {phishing, legitimate}  |  Post-hoc collapse: spam + phishing logits summed to form threat probability  |  Go wrapper port = 8086  |  Python inference port = 8001  |  P50 CPU latency = 80.7 ms  |  Phishing Recall = 98.60%

1. Service Role in the Pipeline

SVC-06 is the natural language processing engine of the CyberSiren threat detection pipeline. It receives email subject and body text, runs the text through a fine-tuned DistilBERT ONNX model, and returns a classification verdict together with derived signals (content risk score, intent labels, urgency score, obfuscation detection).

Attribute Value
Pipeline positionCalled by SVC-07 (aggregator) in parallel with SVC-03 (URL analysis) and SVC-04/05 (header/attachment). All results are aggregated into the final verdict.
CallerSVC-07 aggregator — HTTP POST /predict with 10-second timeout.
InputEmail subject + body plain text. Optional HTML body accepted; stripped to plain text internally.
OutputClassification (phishing | legitimate), confidence [0–1], phishing probability [0–1], content risk score [0–100], intent labels (multi-label list), urgency score [0–1], obfuscation flag.
DeploymentSingle Docker container running Python FastAPI (port 8001) + Go HTTP wrapper (port 8086) with a shared entrypoint script.

2.2 Python Inference Initialization Sequence (background thread)

1
Lifespan hook — FastAPI async lifespan context manager creates a placeholder NLPInferenceEngine (no model loaded) to allow /status progress polling immediately.
2
Config load [5%] — Reads nlp/config.json: label_map, intent_taxonomy, max_length=256, head_tokens=64, tail_tokens=190, input format template, temperature=1.0. Stage: "loading_config".
3
Tokenizer load [15%] — Attempts DistilBertTokenizerFast.from_pretrained("./tokenizer/") (local bundled tokenizer files). Falls back to HuggingFace hub download (distilbert-base-uncased) if directory absent. Stage: "loading_tokenizer".
4
Model check [30%] — Checks nlp/onnx/model_int8.onnx exists and is >1 KB. If missing or placeholder, sets loading_stage="model_file_missing" and aborts. Stage: "checking_model".
5
ONNX session [45%] — Determines cache path: model_int8_opt.onnx alongside the model file. Cache hit (file exists): loads from cache (~5–15 s). Cache miss (first run): loads original model and sets optimized_model_filepath to write the optimized graph (~60–130 s). ORT settings: ORT_ENABLE_ALL optimization, CPUExecutionProvider, intra-op threads = ORT_INTRA_OP_THREADS env var (default: auto), inter-op threads = 1, sequential execution mode, log severity = 3 (silent). Stage: "loading_onnx" or "loading_cached_onnx".
6
Warmup [90%] — Runs inference on a dummy batch (shape 1×32) with zero token IDs to trigger JIT graph compilation. Prevents first real request from paying compilation overhead. Stage: "warming_up".
7
Ready [100%] — model_ready = True set LAST (after session and warmup are both complete). Race safety: predict() checks not self.model_ready or self.session is None. In the error path, model_ready = False is set BEFORE session = None. Stage: "ready". GET /healthz now returns 200; POST /predict now returns 200 instead of 503.

3. Text Preprocessing Pipeline

All raw email input passes through the preprocessing pipeline before tokenization. Preprocessing is implemented in nlp/inference.py _preprocess() and its helpers.

3.1 HTML Stripping

3.2 Unicode Normalization

OperationDetailRationale
NFKC decompositionunicodedata.normalize("NFKC", text)Collapses compatibility variants and ligatures. Cyrillic 'а' (U+0430) → Latin 'a'. Fullwidth chars → ASCII equivalents.
Zero-width removalStrip U+200B, U+200C, U+200D, U+FEFF, U+00AD, U+FE00–U+FE0FPrevents tokenizer fragmentation by adversarial ZWS injection. These characters are invisible to readers but disrupt token boundaries.

3.3 Obfuscation Detection

Method: Compare raw concatenated input text against the NFKC-normalized version. If NFKC(raw) != raw OR if zero-width characters were present in the raw text, obfuscation_detected = True. This flag is returned as part of the prediction response and can be used by SVC-07 to boost the risk score.

3.4 Input Composition

The normalized subject and body are concatenated into a single string using the template from config.json.input_format:

"Subject: {normalized_subject}\n\nBody: {normalized_body}"

This exposes the subject as a named field, enabling the model to weight subject-line signals (urgency cues, sender impersonation, brand names) appropriately. The format is identical between training and inference — critical for consistency.

4. Head-Tail Tokenization

DistilBERT has a maximum input capacity of 512 tokens. The max_length configuration key sets the effective limit to 256 tokens (including CLS and SEP). Emails exceeding this are truncated using a head-tail strategy rather than naive head-only truncation.

4.1 Rationale for Head-Tail

RegionTokens AllocatedSemantic Content
Head64Opening salutation, urgency lure, brand impersonation, subject-line hook. Phishing emails frontload their social engineering.
Tail190Call-to-action link, "verify your account" button, legal disclaimer, obfuscated unsubscribe. Phishing CTAs and obfuscation artefacts cluster in footers.
CLS + SEP2Special tokens: [CLS]=101 prepended, [SEP]=102 appended. Total: 256.

4.2 Encoding Algorithm

1
Tokenize — tokenizer.encode(text, add_special_tokens=False, truncation=False) → list of integer token IDs. No truncation at this stage; the full token sequence is produced first.
2
Content budget — keep = max_length - 2 = 254 content slots. If len(ids) ≤ keep, use all tokens with no truncation.
3
Head-tail split — head = min(head_tokens, keep) = 64; tail = min(tail_tokens, keep) = 190. If head+tail ≠ keep, head is extended to fill the remainder. Result: ids = ids[:head] + ids[-tail:].
4
Wrap — Prepend CLS token (101) and append SEP token (102). Attention mask = all-ones (no padding; all tokens attended to). Shape: input_ids: [1 × seq_len], attention_mask: [1 × seq_len] where seq_len ≤ 256.

5.3 Graph Optimization Cache (ORT Cache Fix)

Implementation detail: The standard pattern (sess_opts.optimized_model_filepath = cache_path) only tells ORT where to write the optimized graph — it does not cause subsequent runs to load from the cache. To benefit from the cache on runs after the first, the code explicitly checks whether the cache file exists at startup. If it does, the cache file path is passed as the model_path to InferenceSession (and optimized_model_filepath is NOT set). This reduces cold-start time from 60–130 s to 5–15 s on subsequent container launches.

# Input tensors
input_ids_array    = np.array([[101, ...token_ids..., 102]], dtype=np.int64)  # shape (1, seq_len)

    input_feed={

        "attention_mask": attention_mask,
    }
)

logits = outputs[0][0]   # shape (3,) → [legitimate_logit, spam_logit, phishing_logit]

5.5 Temperature Scaling

Raw logits are divided by temperature (default: 1.0) before softmax. Temperature was fitted on the validation set during training; 1.0 indicates no scaling was required for calibration. The key enables future recalibration without model retraining.

probs = softmax(logits / temperature)   # probs.shape = (3,), sums to 1.0

6. Post-Hoc Classification Collapse

Background: The deployed INT8 model has a 3-class output head (legitimate / spam / phishing). After ONNX INT8 dynamic quantization, the model's softmax probabilities are poorly calibrated between class 1 (spam) and class 2 (phishing): phishing emails reliably score higher in the spam bucket than the phishing bucket. This is a known INT8 quantization artefact — per-tensor quantization loses fine-grained calibration between adjacent classes. The correct long-term fix is retraining with calibration data. The runtime fix applied here is post-hoc class collapse.

6.1 Collapse Logic

leg_prob    = probs[0]               # P(legitimate)
threat_prob = probs[1] + probs[2]    # P(spam) + P(phishing) = P(not legitimate)

if threat_prob > leg_prob:
    classification = "phishing"
    confidence     = threat_prob
else:
    classification = "legitimate"
    confidence     = leg_prob

phishing_probability  = threat_prob              # 0.0–1.0
content_risk_score    = round(threat_prob * 100) # 0–100

6.2 Mathematical Interpretation

InterpretationDetail
Binary equivalentEquivalent to training a binary classifier on {legitimate} vs {spam ∪ phishing}. Any "not legitimate" signal is treated as threat-positive.
Threshold-freeNo probability threshold is applied. The decision is purely argmax({leg_prob, threat_prob}). The phish_threshold config key is present but unused in this implementation.
Confidence semanticsConfidence = max(leg_prob, threat_prob). Always ≥ 0.5 by definition. A value of 0.5 indicates maximum uncertainty between the two classes.
Spam referencesThe label "spam" does not appear in any API response. The model's internal spam class logit contributes to phishing_probability and content_risk_score but is not surfaced separately.

6.3 Label Map (config.json)

The label_map in config.json reflects the model's training-time 3-class schema: {"0": "legitimate", "1": "spam", "2": "phishing"}. This is not the API output schema — it documents the model head's class indices. The API output is always binary (phishing / legitimate).

7.2 Detection Algorithm

def _detect_intent(text: str, classification: str) -> list[str]:
    # Legitimate emails always get benign_notification — no keyword scan needed
    if classification == "legitimate":
        return ["benign_notification"]

    # Phishing: scan all intent categories (except benign_notification)
    text_lower = text.lower()
    matched = [
        intent for intent, patterns in _INTENT_PATTERNS.items()
        if intent != "benign_notification"
        and any(re.search(pattern, text_lower) for pattern in patterns)
    ]

    # Fallback: if no pattern matched, default to credential_harvest
    return matched if matched else ["credential_harvest"]
Multi-label: An email can match multiple intent categories simultaneously (e.g., ["credential_harvest", "urgency_threat", "impersonation"]). All matched intents are returned. Order is deterministic (insertion order of _INTENT_PATTERNS dict, Python 3.7+).

# Start Python inference service in background

# Start Go HTTP wrapper in foreground (PID 1)

The Go wrapper is the foreground process (PID 1) — container lifecycle is tied to it. Python runs as a detached background subprocess. If Python crashes, the Go wrapper returns 503 on subsequent /predict calls until the process is restarted.

16. Design Decisions

DecisionRationale
Dual-service architecture (Go + Python) Same pattern as SVC-03 (URL Analysis). Go handles HTTP, observability, graceful shutdown, and future Kafka integration. Python handles ML inference (ONNX Runtime, HuggingFace Transformers, scikit-learn dependencies). Separation of concerns; Go binary stays lightweight.
Post-hoc spam+phishing collapse INT8 dynamic quantization loses calibration between adjacent classes. The deployed model's spam and phishing logits are swapped relative to FP32. Rather than modify the model (retraining required), the spam and phishing probabilities are summed into a single "threat" probability. This is mathematically equivalent to a binary classifier and requires no model changes. See §6.
ORT cache load fix opts.optimized_model_filepath = path writes the cache but does not load from it. On subsequent runs, the cached path is passed as model_path directly and optimized_model_filepath is not set. This cuts cold-start time from 60–130 s to 5–15 s after first run.
model_ready race safety Python's GIL makes individual attribute assignments atomic, but multi-line state transitions are not. Fix: model_ready = True is set as the very last statement after session and warmup are both complete. In the error path, model_ready = False is set before session = None. predict() guards with not self.model_ready or self.session is None.
Prometheus label cardinality bound Unbounded label sets cause Prometheus memory exhaustion. The classification label is whitelisted to {phishing, legitimate, unknown}. Any unexpected value from the Python service is collapsed to "unknown" before the counter increment.
Zero retries in Go HTTP client A 503 from the Python service means the model is not loaded — not a transient error. Retrying would queue requests during a known-unavailable period. The Go wrapper surfaces the 503 immediately so callers can fail fast or poll /status.
wc -c for model size check stat -c%s is GNU-only (Linux). macOS uses BSD stat (stat -f%z). wc -c < file is POSIX-portable and works on both Linux and macOS developer machines.
Rule-based intent detection (not trained head) Annotated training data for the auxiliary intent head is not yet available. The rule-based pattern matcher covers the 11-intent taxonomy and correctly classifies all demo email examples. It is explicitly documented as a limitation and a known improvement path.
head=64, tail=190 token split Phishing emails front-load lures and urgency (64 head tokens sufficient). The call-to-action, malicious link, legal boilerplate, and obfuscation artefacts cluster at the end (190 tail tokens). Middle content is sacrificed; it typically contains filler text with low discriminative signal.
max_length=256 (not 512) Halves memory bandwidth and compute for the attention mechanism (quadratic in sequence length). At 256 tokens: head=64 + tail=190 + CLS+SEP=2. P50 latency = 80.7 ms — well within the 150 ms target. Full 512-token inference would ~4× the attention compute for marginal accuracy gain.

17. Limitations

#LimitationImpactMitigation / Next Step
1 Spam/phishing calibration gap in INT8 model. The 3-class probability estimates for spam and phishing are unreliable relative to each other after INT8 quantization. Post-hoc collapse masks the issue for binary classification but the "spam" class information is lost. Multi-class reporting is not available. Retrain with per-class quantization calibration data or export FP32 model for calibration before INT8 conversion. Post-hoc collapse is the runtime mitigation.
2 Intent detection is rule-based (keyword regex). No learned intent head. Novel phishing intent patterns not covered by the keyword dictionary will be undetected or misclassified. Annotate training data with intent labels. Train auxiliary intent classification head as specified in NLP-SPEC-v1.0 §3.3.
3 top_tokens is always an empty array. Explainability (LIME/SHAP token attribution) is offline only. No runtime explanation is returned. Implement LIME with cached perturbations or integrate SHAP as an async background task with result caching.
4 Model trained on 2004–2025 datasets. LLM-crafted phishing (GPT-4, Claude) is underrepresented. AI-generated phishing that mimics legitimate language closely may evade the model. Expand training data with AI-generated phishing corpora (D4 already included at small scale). Periodic retraining.
5 Python process not supervised by Go wrapper. If the Python inference service crashes, Go wrapper returns 503 on all /predict calls. No automatic restart. Use a process supervisor (supervisord) or restructure entrypoint to restart Python on failure. Docker restart policy does not handle internal process crashes.
6 No authentication on /predict endpoint. Service is designed as an internal pipeline component. Direct external exposure would allow unauthenticated inference calls. Deploy behind internal network boundary (Kubernetes ClusterIP or Docker Compose internal network). Add API key or JWT auth if external exposure is required.
7 Single-threaded ONNX inference (one request at a time). Under concurrent load, requests queue behind the 80–100 ms inference time. P99 latency degrades linearly with queue depth. Run multiple Uvicorn worker processes (--workers N) with one ONNX session per worker. Each worker consumes ~67 MB model memory.
