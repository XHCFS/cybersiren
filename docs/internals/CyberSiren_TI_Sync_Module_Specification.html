CYBERSIREN — THREAT INTELLIGENCE SYNC MODULE SPECIFICATION

Document: TI-SPEC-v1.1 Date: 2026-04-10 Revised: MalwareBazaar feed (GAP-1/2/3), hash routing separation, ThreatFox multi-type IOC, hash cache TTL config Scope: SVC-11 TI Sync — Feed ingestion, repository layer, Valkey cache, runner lifecycle Classification: Internal / Graduation Project
Context: SVC-11 (ti-sync) is a standalone Go service that periodically fetches threat-intelligence indicators from external feeds, upserts them into PostgreSQL, refreshes materialized views, and rebuilds two Valkey caches: a domain blocklist cache (for URL analysis) and a malware hash cache (for attachment analysis). The service exposes Prometheus metrics and OpenTelemetry traces. It does not participate in the Kafka pipeline — it is a scheduled background synchronizer.
Live run results (2026-04-10, post-PR): 2,678 active indicators across 4 feeds — OpenPhish (300 URLs), URLhaus (1,946 URLs), ThreatFox (432 mixed: URLs + domains + IPs), MalwareBazaar (30 malicious hashes in attachment_library; 0 rows in ti_indicators by design). 38 ti_hash:* keys and ~400 domain keys cached in Valkey.

1. Service Architecture

1.1 Bootstrap Order

1
Bootstrap logger — zerolog to stderr, RFC 3339 Nano timestamps. Unstructured until config is loaded.
2
config.Load() — koanf: process env → .env file → config.yaml → defaults. Env prefix: CYBERSIREN_. Double underscore maps to nested struct (CYBERSIREN_DB__PASSWORD → cfg.DB.Password). Calls cfg.Validate() — fatal on missing required fields.
3
Signal context — signal.NotifyContext(ctx, SIGINT, SIGTERM). Cancels the root context on OS signal to trigger graceful shutdown.
4
Tracing init — OTLP exporter if CYBERSIREN_JAEGER_ENDPOINT is set, otherwise noop tracer. Global tracer provider installed via otel.SetTracerProvider.
5
Metrics server — Prometheus HTTP server on CYBERSIREN_METRICS_PORT (default 9090). Exposes /metrics. Runs as background goroutine.
6
PostgreSQL pool — pgx/v5 pool with OTel instrumentation. Verifies connectivity with ping.
7
Valkey client — valkey-go client. Verifies connectivity with PING.
8
Repository + cache init — repository.NewTIRepository(pool, log, registry) and valkey.NewTICache(client, repo, log, registry).
9
Feed discovery — queries feeds table for enabled rows (enabled=true, deleted_at IS NULL). Builds feed implementations via name switch. Skips unknown names with a warning.
10
Runner.Start(ctx, interval) — executes initial sync immediately, then ticks on CYBERSIREN_SYNC_INTERVAL_SECONDS (default 3,600 s / 1 h). Fatal if no feeds are enabled.

4. Runner — Sync Lifecycle

4.1 SyncAll() Flow

Called on startup and on each tick. Processes feeds sequentially. Per-feed errors do not abort remaining feeds.

1
FOR EACH feed: Call feed.Fetch(feedCtx). On error: log error, increment feed_sync_errors_total{feed_name} label, update feeds.last_fetched_at, continue to next feed.
2
BulkUpsertIndicators() — Passes all non-hash indicators to the repository. On error: log, continue. Success: log inserted/updated/deactivated counts.
3
extractHashIndicators() → BulkUpsertMalwareHashes() — Filters indicators with IndicatorType == "hash". If any found, calls BulkUpsertMalwareHashes() to write to attachment_library. For mixed feeds (e.g. ThreatFox returning both hash and non-hash), hash upsert errors are logged but non-fatal. For hash-only feeds (e.g. MalwareBazaar where len(nonHashIndicators) == 0), hash upsert failure is treated as fatal — the feed is counted as failed and the cycle continues to the next feed.
3a
DeactivateStaleIndicators() — hash-only feeds only — When a feed returned only hash indicators (len(hashIndicators) > 0 && len(nonHashIndicators) == 0), BulkUpsertIndicators received an empty slice and early-returned without running its internal deactivation. An explicit DeactivateStaleIndicators(feedID) call is made after a successful hash upsert to clean up any stale rows in ti_indicators for that feed (e.g. from runs before hash routing was introduced). Deactivation errors are logged but do not fail the feed.
4
UpdateFeedLastFetched() — Stamps feeds.last_fetched_at = NOW() for the feed row, regardless of fetch success or failure.
5
RefreshDomainCache() — Rebuilds ti_domain:{domain} Valkey keys from all active domain indicators. Best-effort; cache failure does not abort the sync.
6
RefreshHashCache() — Rebuilds ti_hash:{sha256} Valkey keys from all malicious attachment_library rows. Best-effort.
7
RefreshAllMaterializedViews() — Runs REFRESH MATERIALIZED VIEW CONCURRENTLY on all TI-related views. Fatal error if this fails (only failure that returns an error from SyncAll).

5.2 BulkUpsertIndicators — Upsert Semantics

Conflict key: (feed_id, indicator_type, indicator_value). On conflict: last_seen updated to NOW, threat_tags merged via ARRAY DISTINCT union (sorted), risk_score = GREATEST of existing and new. first_seen and created_at are never overwritten. After each successful upsert batch, rows not seen since the start of the current sync are deactivated (is_active = FALSE).
FieldConflict Resolution
last_seenAlways updated to EXCLUDED (current fetch time)
threat_tagsARRAY DISTINCT union of existing + incoming, sorted alphabetically
risk_scoreGREATEST(existing, incoming) — score never decreases
first_seenNever overwritten — preserves original discovery time
is_activeSet TRUE on any insert/update. Stale rows set FALSE post-batch.
raw_metadataNot updated on conflict — original JSON preserved

5.3 BulkUpsertMalwareHashes — Upsert Semantics

Conflict key: sha256 (UNIQUE constraint on attachment_library). On conflict: is_malicious set TRUE, risk_score = GREATEST, threat_tags merged as above, updated_at = NOW(). Hashes written here represent files confirmed malicious by a TI feed; they may later be matched by SVC-05 attachment analysis.

9. Design Decisions

DecisionRationale
Hash indicators routed to attachment_libraryThe TI indicator pipeline is optimized for URL/domain/IP blocklist lookups. Hashes represent file identity and belong alongside uploaded attachment records in attachment_library. Routing them there enables SVC-05 to join TI data without a separate table lookup and keeps ti_indicators focused on network-based IOCs.
Sequential feed processingEach feed is fetched and upserted sequentially within SyncAll. This avoids connection pool exhaustion under load, simplifies error isolation, and keeps the sync cycle predictable. Feeds are fast enough (all complete in <10 s total) that parallelism provides no meaningful benefit.
Conflict: GREATEST risk_scoreRisk scores should never decrease. If a feed previously reported score 90 and now reports 70 (e.g., due to API changes), the historical high is preserved. Operators can manually decrease via the admin API.
Conflict: tag union not replacementTags from multiple sync cycles accumulate. A domain tagged ["phishing"] in one cycle and ["malware", "botnet"] in the next retains all three tags. This preserves historical context from feeds that rotate their tag classifications.
is_active deactivationIndicators not seen in the current sync cycle are marked is_active = FALSE rather than deleted. This allows downstream services to distinguish between "currently blocklisted" and "was blocklisted" without losing the historical record.
Valkey domain cache TTL = 1 hourDomain cache TTL is fixed at 3,600 s. With a 1-hour default sync interval, domain cache is fully rebuilt each cycle before prior keys expire. Staleness is bounded to one sync window.
Valkey hash cache TTL = 2 hours (configurable)Hash cache TTL defaults to 7,200 s (2 × the default sync interval). The validation rule TIHashCacheTTLSeconds ≥ SyncIntervalSeconds ensures hash keys never expire between sync runs. The 2× default gives headroom for slower hash refreshes. The cache is additive: removed hashes persist until TTL expiry (see Limitations §10.7).
Feed URL from DB at runtimeThe URL stored in the feeds.url column overrides the hardcoded default constant. This allows feed URLs to be updated via the admin API without a service redeployment. PhishTank uses a template URL; the API key is still injected from config, not the DB.
Atomic overlap guardatomic.Bool cycleRunning prevents two sync cycles from running simultaneously if the previous cycle overruns the interval. This is safer than a mutex approach because the check is non-blocking and a skipped tick is recoverable on the next tick.
MalwareBazaar Auth-Key requirementAs of 2025, abuse.ch requires authentication for all MalwareBazaar API calls (previously only the enhanced tier). The implementation sends the key as an Auth-Key HTTP header matching the pattern used by ThreatFox. A missing key logs a warning but does not prevent service startup; the feed will fail at fetch time with HTTP 401.
ThreatFox ip:port stripping + IPv6 safetyThreatFox reports C2 servers as 1.2.3.4:6000 (ip:port format). Port stripping via net.SplitHostPort() is applied only for bracketed IPv6 ([::1]:80) or single-colon IPv4 (1.2.3.4:80). Bare IPv6 addresses (multiple colons, no brackets) bypass SplitHostPort to avoid mis-parsing. CyberSiren's ti_indicators has no port field — indicators represent hosts, not endpoints. Port context is preserved in raw_metadata.

10. Limitations

#LimitationImpactMitigation
1No parallel feed fetching.Total sync time = sum of all feed fetch times. Currently ~8 s for 4 feeds.Acceptable at current scale. Parallel fetch can be added if feed count grows significantly.
2No per-feed scheduling.All feeds sync on the same interval (default 1 h). High-frequency feeds (URLhaus updates every 5 min) are under-sampled.Use a shorter global interval or add per-feed cron support in a future iteration.
3No distributed leader election.Running multiple instances of svc-11 simultaneously will cause duplicate upsert contention. Upsert semantics are idempotent, so data integrity is maintained, but it wastes API calls and DB write pressure.
4Hash cache holds entire malicious set in memory per refresh.If attachment_library grows to millions of entries, the refresh will hold a large slice in RAM. Batch streaming would be needed.Current scale (hundreds to low thousands of entries) is well within limits.
5PhishTank feed disabled unless API key is provided.PhishTank is the community-verified source with highest precision for known phishing URLs. Without it, URL coverage relies on OpenPhish + URLhaus.Register at phishtank.org, set CYBERSIREN_FEED_PHISHTANK_API_KEY.
6Raw metadata not updated on indicator conflict.If a feed changes the payload for an existing IOC (new confidence level, additional fields), the stored raw_metadata jsonb reflects the first-seen payload only.Risk score and tags are updated on conflict. For full payload history, store in an event log (not implemented).
7Hash cache (ti_hash:* keys) is additive — stale keys are not deleted on refresh.If a hash is removed from attachment_library (e.g. false-positive correction), its Valkey key persists until TTL expiry (default 2 h). SVC-05 may still return a hit for that hash within the TTL window.Acceptable for a security context where false positives are preferable to false negatives. TTL bounds the staleness window. A future iteration could use a generation-swap pattern (write to a new key prefix, then rename) to achieve atomic, non-stale rebuilds.
